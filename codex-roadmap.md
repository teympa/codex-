# Codex向け実装ロードマップ

以下は、4フェーズ案を `Codex` で実装しやすい単位に落としたロードマップです。  
方針は `先に情報の置き場を固定 -> 入力経路を増やす -> 制作ワークフローを自動化 -> 外部公開へ広げる` です。

## 現在地

2026-04-21 時点では、以下まで進んでいます。

- Phase 1 の基盤文書整備
- GitHub Issue `#1` から `#5` の起票
- `context.md` 運用開始
- 正本ルールの固定
- GitHub と Notion の同期方針固定
- Notion の 5 DB 作成と初期データ投入
- Discord Bot MVP 実装
- `/codex` `/codex-status` `/codex-confirm` の実装
- Discord 実行ログ保存
- Discord allowlist ベースのアクセス制御
- 確認待ちアクションの永続化

次の優先タスクは `GitHub / Notion の Tasks 同期半自動化` です。

## Phase 1 基盤整備

目的は、全作業の保存先と参照先を固定することです。

### 1. Notion情報設計

- 仕様書DBを作る
- 企画メモDBを作る
- タスク/進行管理DBを作る
- ナレッジDBを作る
- 各DBの必須プロパティを定義する
- ページテンプレートを作る

### 2. GitHub運用設計

- リポジトリ構成を決める
- Issueテンプレートを作る
- PRテンプレートを作る
- ラベル設計を決める
- Milestone運用ルールを決める
- ブランチ命名規則を決める

### 3. `context.md` 設計

- 現在進行中プロジェクト一覧
- 今週の優先事項
- 継続中の判断ログ
- 重要リンク集
- 定例運用ルール
- 次回Codex起動時の引き継ぎ欄

### 4. Codex作業導線の整備

- `docs/` 配下の構造を決める
- `context.md` の更新ルールを決める
- NotionとGitHubのどちらを正本にするか決める
- MCP利用対象を整理する
- 日次/週次の更新フローを決める

### 5. Phase 1 完了条件

- Notionに主要DBが揃っている
- GitHubテンプレ/ラベルが運用可能
- `context.md` が1回の作業で更新される
- Codexが迷わず参照先を判断できる

## Phase 2 リモート操作

目的は、スマホから最低限の運用を回せるようにすることです。

### 1. Discord操作設計

- Discord Botを作る
- コマンド一覧を定義する
- 権限管理を決める
- ログ保存先を決める
- エラー時の通知方法を決める

### 2. コマンド体系の実装

- `/codex`
- `/codex-status`
- `/codex-confirm`

補足:

- `/codex` は自然文で Codex に指示する入口
- `/codex-status` は `context.md`、GitHub、Notion `Tasks` を集約して返す
- `/codex-confirm` は変更系指示の確認待ち token を承認する
- `/memo` `/task` `/schedule` などの専用コマンドは、必要なら次段階で追加する

### 3. バックエンド接続

- Discordから Codex 実行ランナーへ指示を渡す
- `/codex-status` で GitHub 状態を返す
- `/codex-status` で Notion `Tasks` 集計を返す
- 実行履歴を JSONL で残す
- 確認待ちアクションを再起動後も復元できるようにする

### 4. スマホ運用向け整備

- 短い自然文でも動く入力形式を決める
- 定型文を用意する
- 誤操作防止の確認フローを入れる
- 重要操作は承認制にする
- 利用できる guild / channel / user を絞る

### 5. Phase 2 完了条件

- スマホから Codex に自然文指示を出せる
- 読み取り系と変更系を安全に振り分けられる
- `/codex-status` で現在状況を確認できる
- 実行ログを後から追える
- 再起動後も確認待ちアクションを継続できる

## Phase 3 ゲーム開発ワークフロー

目的は、企画から実装、素材生成までの流れを半自動化することです。

### 1. プロジェクト雛形の整備

- ゲーム企画用テンプレートを作る
- 技術設計テンプレートを作る
- タスクリスト雛形を作る
- リポジトリ初期構成をテンプレ化する

### 2. GitHubとNotionの同期

- Issue作成時にNotionタスクを作る
- ステータス変更時にNotion反映する
- PR作成時にレビュー待ちへ更新する
- 完了時にナレッジへ記録する

### 3. 企画書自動生成

- アイデア入力テンプレートを作る
- 企画書Markdownテンプレートを作る
- Notionページ自動生成フローを作る
- 企画からIssue群へ分解する仕組みを作る

### 4. ComfyUI連携

- ComfyUI環境を整える
- プロンプト管理場所を決める
- アセット生成ワークフローを定義する
- 出力ファイルの保存規則を決める
- 採用/不採用の記録方法を決める

### 5. 開発支援の自動化

- 日次進捗サマリ生成
- 週次レビュー生成
- 仕様変更差分の要約
- ドキュメントの更新提案
- テスト/確認タスクの自動起票

### 6. Phase 3 完了条件

- 新規企画から雛形一式を自動生成できる
- IssueとNotionが最低限同期する
- ComfyUI成果物を整理して保管できる
- 進捗と判断履歴が追跡できる

## Phase 4 公開・発信

目的は、内部情報を見せる形に変換し、発信までつなぐことです。

### 1. ライブWebダッシュボード

- 表示対象データを決める
- 認証要否を決める
- 進捗、Issue、更新履歴を表示する
- モバイル対応する
- 更新頻度を決める

### 2. 外部向けドキュメントサイト

- 公開可能な仕様書だけ抽出する
- 公開/非公開フラグを持たせる
- Webから閲覧可能にする
- 必要なら簡易編集機能をつける
- 更新履歴を残す

### 3. note記事パイプライン

- 記事ネタ収集導線を作る
- 週次でネタ候補をまとめる
- 下書きテンプレートを作る
- 開発ログから記事案を生成する
- 投稿前レビュー工程を入れる

### 4. ゲーム業界特化ナレッジ発信

- 業界ニュースの保存先を決める
- 調査メモの形式を決める
- note用の要約フォーマットを作る
- 発信テーマ別にタグ管理する

### 5. Phase 4 完了条件

- ダッシュボードで進捗が見える
- 公開可能な情報をWebで見せられる
- note記事の下書き生成が回る
- 調査メモが発信素材に変換できる
