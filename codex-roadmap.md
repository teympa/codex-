# Codex向け実装ロードマップ

以下は、4フェーズ案を `Codex` で実装しやすい単位に落としたロードマップです。  
方針は `先に情報の置き場を固定 -> 入力経路を増やす -> 制作ワークフローを自動化 -> 外部公開へ広げる` です。

## Phase 1 基盤整備

目的は、全作業の保存先と参照先を固定することです。

### 1. Notion情報設計

- 仕様書DBを作る
- 企画メモDBを作る
- タスク/進行管理DBを作る
- ナレッジDBを作る
- 各DBの必須プロパティを定義する
- ページテンプレートを作る

### 2. GitHub運用設計

- リポジトリ構成を決める
- Issueテンプレートを作る
- PRテンプレートを作る
- ラベル設計を決める
- Milestone運用ルールを決める
- ブランチ命名規則を決める

### 3. `context.md` 設計

- 現在進行中プロジェクト一覧
- 今週の優先事項
- 継続中の判断ログ
- 重要リンク集
- 定例運用ルール
- 次回Codex起動時の引き継ぎ欄

### 4. Codex作業導線の整備

- `docs/` 配下の構造を決める
- `context.md` の更新ルールを決める
- NotionとGitHubのどちらを正本にするか決める
- MCP利用対象を整理する
- 日次/週次の更新フローを決める

### 5. Phase 1 完了条件

- Notionに主要DBが揃っている
- GitHubテンプレ/ラベルが運用可能
- `context.md` が1回の作業で更新される
- Codexが迷わず参照先を判断できる

## Phase 2 リモート操作

目的は、スマホから最低限の運用を回せるようにすることです。

### 1. Discord操作設計

- Discord Botを作る
- コマンド一覧を定義する
- 権限管理を決める
- ログ保存先を決める
- エラー時の通知方法を決める

### 2. コマンド体系の実装

- `/memo`
- `/task`
- `/schedule`
- `/expense`
- `/idea`
- `/status`

### 3. バックエンド接続

- DiscordからNotionへ書き込む
- DiscordからGitHub Issueを起票する
- スケジュール情報を保存する
- 家計簿データを保存する
- 実行履歴を残す

### 4. スマホ運用向け整備

- 短い自然文でも動く入力形式を決める
- 定型文を用意する
- 誤操作防止の確認フローを入れる
- 重要操作は承認制にする

### 5. Phase 2 完了条件

- スマホからメモ追加ができる
- スマホからタスク登録ができる
- スマホから予定登録ができる
- スマホから支出記録ができる
- 実行ログを後から追える

## Phase 3 ゲーム開発ワークフロー

目的は、企画から実装、素材生成までの流れを半自動化することです。

### 1. プロジェクト雛形の整備

- ゲーム企画用テンプレートを作る
- 技術設計テンプレートを作る
- タスクリスト雛形を作る
- リポジトリ初期構成をテンプレ化する

### 2. GitHubとNotionの同期

- Issue作成時にNotionタスクを作る
- ステータス変更時にNotion反映する
- PR作成時にレビュー待ちへ更新する
- 完了時にナレッジへ記録する

### 3. 企画書自動生成

- アイデア入力テンプレートを作る
- 企画書Markdownテンプレートを作る
- Notionページ自動生成フローを作る
- 企画からIssue群へ分解する仕組みを作る

### 4. ComfyUI連携

- ComfyUI環境を整える
- プロンプト管理場所を決める
- アセット生成ワークフローを定義する
- 出力ファイルの保存規則を決める
- 採用/不採用の記録方法を決める

### 5. 開発支援の自動化

- 日次進捗サマリ生成
- 週次レビュー生成
- 仕様変更差分の要約
- ドキュメントの更新提案
- テスト/確認タスクの自動起票

### 6. Phase 3 完了条件

- 新規企画から雛形一式を自動生成できる
- IssueとNotionが最低限同期する
- ComfyUI成果物を整理して保管できる
- 進捗と判断履歴が追跡できる

## Phase 4 公開・発信

目的は、内部情報を見せる形に変換し、発信までつなぐことです。

### 1. ライブWebダッシュボード

- 表示対象データを決める
- 認証要否を決める
- 進捗、Issue、更新履歴を表示する
- モバイル対応する
- 更新頻度を決める

### 2. 外部向けドキュメントサイト

- 公開可能な仕様書だけ抽出する
- 公開/非公開フラグを持たせる
- Webから閲覧可能にする
- 必要なら簡易編集機能をつける
- 更新履歴を残す

### 3. note記事パイプライン

- 記事ネタ収集導線を作る
- 週次でネタ候補をまとめる
- 下書きテンプレートを作る
- 開発ログから記事案を生成する
- 投稿前レビュー工程を入れる

### 4. ゲーム業界特化ナレッジ発信

- 業界ニュースの保存先を決める
- 調査メモの形式を決める
- note用の要約フォーマットを作る
- 発信テーマ別にタグ管理する

### 5. Phase 4 完了条件

- ダッシュボードで進捗が見える
- 公開可能な情報をWebで見せられる
- note記事の下書き生成が回る
- 調査メモが発信素材に変換できる

## 優先順位

実装順はこの並びがおすすめです。

1. Notion DB設計
2. GitHubテンプレ/ラベル整備
3. `context.md` 整備
4. Discord Bot最小版
5. GitHub-Notion同期
6. 企画書テンプレ自動生成
7. ComfyUI連携
8. Webダッシュボード
9. note記事パイプライン

## 最初の4週間プラン

- 1週目: Notion DB、GitHub運用、`context.md`
- 2週目: Codex運用ルール、テンプレート、基本自動化
- 3週目: Discord Bot最小版、メモ/タスク/予定登録
- 4週目: GitHub-Notion同期、企画書自動生成の初版

## Codexでの実装単位

実際のタスクは、以下の粒度まで分けると進めやすいです。

- `docs/notion-schema.md` を作る
- `docs/github-workflow.md` を作る
- `context.md` の初版を作る
- Issue/PRテンプレを追加する
- Discord Botの雛形を作る
- `/memo` `/task` `/schedule` `/expense` を実装する
- Notion書き込み処理を実装する
- GitHub Issue連携を実装する
- 企画書テンプレ生成スクリプトを作る
- ComfyUI連携スクリプトを作る
- ダッシュボード初版を作る
- note下書き生成スクリプトを作る

このロードマップなら、`Codexにそのまま順番に作業させる形` に落とし込めます。
